## 3. 系统组件与依赖

核心组件：

- 前端：React + TypeScript
- 后端：FastAPI（`backend/app/main.py`）
- 向量库：Chroma（持久化目录 `backend/data/chroma`）
- 检索增强：`intent_filter` + `metadata_enhanced_search` + `reranker`
- 外部依赖：LLM API、ASR 服务、模型下载源

关键路径：

1. 用户输入（文本/语音）进入前端。
2. 前端调用后端 suggestions 接口。
3. 后端执行检索增强与 LLM 生成。
4. 结果返回前端展示。

## 4. 生产配置基线

必须通过环境变量管理，不得把密钥写入代码仓库。

建议基线（按实际环境调整）：

```env
DEEPSEEK_API_KEY=***
DEEPSEEK_BASE_URL=https://api.deepseek.com
DEEPSEEK_MODEL=deepseek-chat

RERANK_ENABLED=true
RAG_TOP_K=15
RAG_SIMILARITY_THRESHOLD=0.35

HOST=0.0.0.0
PORT=8000
```

安全要求：

- 旧维护者个人 token/API Key 必须全部失效。
- 生产密钥统一收敛到企业密钥系统。
- 生产机器最小权限访问（仅开放必要端口与账户）。

## 5. 发布流程（标准 SOP）

### 5.1 发布前检查

- 从 `main` 拉取最新代码，确认目标提交哈希。
- 在预发环境完成冒烟测试：
  - `GET /health`
  - `POST /api/v1/suggestions`
  - `POST /api/v1/suggestions/enhanced`
- 核对环境变量与密钥状态。
- 确认监控看板与告警通道可用。

### 5.2 发布步骤

1. 备份当前可运行版本（代码哈希 + 配置快照）。
2. 发布新版本（重启服务或滚动发布）。
3. 发布后观察 30-60 分钟：
   - 接口成功率
   - P95 延迟
   - fallback 比例
   - ASR 成功率

### 5.3 发布验收标准

- 健康检查通过且持续稳定。
- 关键接口成功率恢复到历史基线。
- 无持续错误告警。

## 6. 回滚流程（应急 SOP）

触发条件（任一满足即可）：

- 核心接口错误率持续升高。
- 延迟大幅劣化且无法快速恢复。
- 建议质量显著退化（人工巡检和评测异常）。

回滚步骤：

1. 切回上一稳定提交。
2. 恢复上一版配置文件或环境变量。
3. 如涉及向量或模型版本变更，执行向量索引重建。
4. 验证关键接口并恢复流量。

## 7. 监控与告警建议

必备指标：

- API 成功率（整体 + 分接口）
- API 延迟（P50/P95/P99）
- LLM 调用失败率与超时率
- fallback 触发率
- ASR 成功率与平均时延
- 机器资源（CPU/内存/磁盘）

建议阈值（可根据基线调整）：

- API 错误率 > 5% 持续 5 分钟报警
- P95 延迟 > 2s 持续 10 分钟报警
- fallback 比例异常升高报警

## 8. 日常运维操作

每日：

- 检查健康接口和核心接口可用性
- 巡检错误日志与告警记录
- 查看资源利用率，确认无容量风险

每周：

- 执行一次 RAG 评测并归档结果
- 复盘告警与故障，更新处理手册
- 校验备份可恢复性

## 9. 故障处理分级与升级路径

故障分级：

- **P0**：全站不可用或核心功能不可用
- **P1**：核心功能可用但显著劣化
- **P2**：非核心异常或局部问题

升级路径（按实际团队补齐）：

1. 值班同学初步止血（降级/回滚/限流）
2. 通知后端负责人
3. 通知前端与产品负责人
4. 必要时触发管理层通报机制
