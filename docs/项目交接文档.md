## 2. 系统是什么

JiSoul 是一个销售场景智能辅助系统，目标是在实时对话或文本输入过程中，基于知识库检索和 LLM 生成，输出可直接使用的话术建议。

当前系统支持两条主能力：

- 文本输入 -> 检索 -> 生成建议。
- 语音输入（ASR）-> 转写 -> 检索 -> 生成建议。

## 3. 架构总览（文字版）

请求链路：

1. 前端发起建议请求（基础版或增强版）。
2. 后端收到请求后执行 RAG 检索。
3. 增强链路会做意图识别、关键词融合、metadata 过滤、priority 加权、可选 reranker 精排。
4. 结果进入 prompt 构建，再调用 LLM 生成建议话术。
5. 返回 suggestions 给前端展示。

技术栈：

- 后端：FastAPI + Pydantic + Chroma + Sentence-Transformers
- 前端：React + TypeScript
- 数据：本地持久化向量库（`backend/data/chroma`）
- 模型：Embedding + Reranker（按配置启用）

## 4. 代码目录导览（新同学先看这个）

核心目录说明：

- `backend/app/main.py`：应用入口，负责启动时初始化模型、向量库、路由注册。
- `backend/app/config.py`：全局配置入口（可被 `.env` 覆盖）。
- `backend/app/api/`：接口层。
  - `routes_suggestion.py`：基础建议接口。
  - `routes_suggestion_enhanced.py`：增强建议接口。
  - `routes_asr.py`：ASR 相关接口。
  - `routes_knowledge.py`：知识库管理接口。
- `backend/app/services/`：业务服务层。
  - `rag_service.py`：基础/主 RAG 服务。
  - `metadata_enhanced_search.py`：增强检索核心。
  - `intent_filter.py`：意图识别与类型过滤。
  - `reranker.py`：重排序模型与触发策略。
  - `rag_eval_feedback.py`：评测反馈能力。
- `backend/app/knowledge/`：知识入库、切片、向量化、存储相关。
- `backend/scripts/`：评测与自动化脚本。
- `frontend/src/components/`：核心交互组件（含录音与转写 UI）。
- `frontend/src/hooks/useRealtimeAsr.ts`：实时 ASR hook。
- `docs/`：部署手册、改造清单、自检报告、交接文档。

## 6. 新人 Day 1 启动指南

### 6.1 环境准备

建议环境：

- Python 3.12+
- Node.js 18+
- npm 9+
- 可访问模型下载源（HuggingFace 或镜像）

### 6.2 后端启动

```bash
cd backend
pip install -r requirements.txt
uvicorn app.main:app --reload --host 0.0.0.0 --port 8000
```

### 6.3 前端启动

```bash
cd frontend
npm install
npm run dev
```

### 6.4 冒烟验证（必须完成）

- 访问 `GET /health`，返回 `status` 正常。
- 调用 `POST /api/v1/suggestions`，确认基础链路可用。
- 调用 `POST /api/v1/suggestions/enhanced`，确认增强链路可用。
- 前端录音后可看到转写内容，并能触发建议生成。

## 7. 环境变量与密钥交接

必须交接的配置项（示例）：

```env
DEEPSEEK_API_KEY=***
DEEPSEEK_BASE_URL=https://api.deepseek.com
DEEPSEEK_MODEL=deepseek-chat
RERANK_ENABLED=true
RAG_TOP_K=15
RAG_SIMILARITY_THRESHOLD=0.35
```

密钥治理要求（强制）：

- 所有旧个人 token/api key 统一下线。
- 使用组织级账号重新签发密钥。
- 新维护人验证后再销毁旧维护人密钥。
- 不允许将 token 写入仓库或脚本明文。

## 8. 数据与模型资产说明

### 8.1 向量库

- 路径：`backend/data/chroma`
- 现状：仓库中已存在一份本地持久化数据
- 风险：体积较大、二进制文件变更频繁，不利于代码审查

建议：

- 明确“是否纳入 Git”的策略（建议迁移到对象存储或制品仓库）。
- 制定每日增量备份 + 每周全量备份策略。

### 8.2 模型缓存

- Embedding 和 Reranker 首次下载时间较长。
- 建议在部署机预热模型，减少服务首次启动延迟。

## 9. 发布与回滚 SOP

### 9.1 发布 SOP（建议）

1. 在预发环境跑冒烟与评测脚本。
2. 对比上一版本关键指标（成功率/耗时/fallback）。
3. 合并并发布。
4. 发布后观察 30~60 分钟核心指标。

### 9.2 回滚触发条件

满足任一条件可回滚：

- 核心接口错误率持续升高。
- 平均延迟超过阈值且无法快速恢复。
- 建议质量明显退化（评测分或人工巡检异常）。

### 9.3 回滚方案

- 代码回滚到上个稳定 tag/commit。
- 重建向量索引（若模型或索引格式变化）。
- 切换配置项关闭 `RERANK_ENABLED` 作为临时止血方案。

## 11. 常见故障与处理手册

- 启动慢：优先检查模型是否首次下载、网络是否可达。
- 检索效果差：先确认向量库是否重建、阈值是否异常修改。
- ASR 不工作：检查麦克风权限、ASR 依赖服务状态、前端日志。
- 建议全 fallback：检查外部 LLM 密钥与网络连通。
- 二进制文件冲突：确认是否误将运行时产物纳入提交。
